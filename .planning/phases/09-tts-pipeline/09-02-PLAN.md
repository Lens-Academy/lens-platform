Create `web_frontend/src/hooks/useAudioPlayback.ts` implementing Web Audio API chunk scheduling (Pattern 3 from research).

**Interface:**
```typescript
export interface UseAudioPlaybackReturn {
  /** Feed an MP3 chunk (ArrayBuffer) for playback */
  playChunk: (mp3Bytes: ArrayBuffer) => Promise<void>;
  /** Stop all playback and reset state */
  stop: () => void;
  /** Whether audio is currently playing */
  isPlaying: boolean;
  /** Number of chunks received so far */
  chunksReceived: number;
  /** Resume AudioContext after user gesture (call on button click) */
  resume: () => Promise<void>;
  /** Current AudioContext state ('suspended' | 'running' | 'closed') */
  contextState: AudioContextState | null;
}

export function useAudioPlayback(): UseAudioPlaybackReturn
```

**Implementation details:**
- Use `useRef` to hold the `AudioContext` instance (lazy-create on first `playChunk` or `resume` call).
- Track `nextStartTime` in a ref for gapless scheduling: each chunk's source node starts at `max(nextStartTime, ctx.currentTime)`, then `nextStartTime = startTime + buffer.duration`.
- `playChunk(mp3Bytes)`: call `ctx.decodeAudioData(mp3Bytes)`, create `AudioBufferSourceNode`, connect to `ctx.destination`, schedule with `start(startTime)`. Set `source.onended` callback to decrement an active source count and update `isPlaying` when all sources finish.
- `stop()`: close the AudioContext, reset nextStartTime, reset state. Create a fresh AudioContext on next use.
- `resume()`: create AudioContext if needed, call `ctx.resume()` to handle autoplay policy. This MUST be called from a user gesture handler (click/tap).
- `isPlaying` state: true when any AudioBufferSourceNode is still playing (track via counter incremented in playChunk, decremented in onended).
- `chunksReceived` state: simple counter incremented in playChunk.
- `contextState` state: updated via `ctx.onstatechange` and on resume/stop.
- Cleanup: use `useEffect` cleanup to close AudioContext on unmount.
- Handle `decodeAudioData` failures gracefully (log warning, skip chunk, do not crash).

**Critical pitfalls to handle (from research):**
- AudioContext autoplay policy: `resume()` must be called from user gesture
- Memory leaks: do NOT keep references to decoded AudioBuffers or source nodes after playback. Let them GC via onended cleanup.
- Do NOT use a single AudioBufferSourceNode for all chunks. Each chunk gets its own node (they are one-shot).

**web_frontend/src/pages/tts-test/+Page.tsx:**

Simple, functional test UI -- NOT production UI, just enough to prove the pipeline works.

Layout:
- Heading: "TTS Pipeline Test"
- Status indicator showing: AudioContext state, WebSocket connection state, TTS availability
- Textarea for entering text to synthesize
- Voice selector dropdown (text input with default "Ashley")
- "Speak" button that initiates TTS
- "Stop" button to halt playback
- Log panel showing events in real-time (connected, chunk received N bytes, playback started, done, errors)

Behavior on "Speak" click:
1. Call `audioPlayback.resume()` to ensure AudioContext is running (user gesture!)
2. Open WebSocket to `/ws/tts` (use `ws://${location.host}/ws/tts` for dev, auto-upgrades in production)
3. Send JSON: `{"text": textareaValue, "voice": selectedVoice}`
4. On binary message: call `audioPlayback.playChunk(event.data)`, log chunk size
5. On text message: parse JSON, check for `{"done": true}` or `{"error": "..."}`, log it
6. On error/close: log it, update status

State management with useState:
- `text` (textarea value)
- `voice` (voice selection, default "Ashley")
- `status` ("idle" | "connecting" | "streaming" | "done" | "error")
- `logs` (array of log strings with timestamps)
- `wsRef` (useRef for WebSocket instance)

Use `useAudioPlayback` hook for all audio handling.

Style with Tailwind -- basic utility classes, nothing fancy. Dark background for log panel (like a console).

WebSocket URL construction: use `${location.protocol === 'https:' ? 'wss:' : 'ws:'}//${location.host}/ws/tts` to handle both dev and production.

Important: The backend API port and frontend port differ (offset by 5000 per CLAUDE.md). The WebSocket must connect to the backend port, not the frontend dev server. Use the same origin logic as the existing API calls -- check how `src/api/modules.ts` or `src/config.ts` constructs API URLs, and use the same base for the WebSocket URL.
